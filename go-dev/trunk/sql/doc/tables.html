association

 Top
 Comments:

association
stores the annotations;
an annotation is essentially a link
between a gene product and an ontology term,
with one or more pieces of evidence
assocdate is a date in YYYYMMDD format
(this is the date the association was made by
the source db providers)
source_db_id is a FK into the db table that indicates
the source of the association; for instance, the association
file may come from SwissProt, but the source of the association
could be MGI
NOTE:
in future we may have more complex statements
about annotations which will require a schema
change, although the current table will probably
be retained as a [materialized] view
"roles"
this table contains a "group" number which is optional
the idea is that associations could be seperated into groups
representing different roles/actions of the gene product
at first this will be unpopulated, as GO currently lumps together
all associations, but this allows the option of statements such as
"product X has function F in cell component C"

assoc_rel

 Top
 Comments:

assoc_rel
NOT POPULATED YET
for "conjunctive" annotations, or "slot-based" annotations.
currently all GO annotations are "disjunctive" (ie unrelated)
this table provides a way of linking together annotations.
for example:
"gene foo has function F during process P in cell component C"
treated as 3 associations: foo/F, foo/P, foo/C
assocs 1 and 2 are related by the type "acts_during" (for example)
assocs 1 and 3 are related by the type "acts_in"
"gene foo has function biosynthesis, acting on interleukin-18"
treated as 2 associations: foo/biosynth, foo/il-18
assocs are related by the type "acts_on" (for example)
the from_id is for the subject (domain); the to_id is for the
object (range). if a gene has function F during process P, then
the from_id is the association to F, and the to_id is the
association to P
the nature of the relationship (aka slot) is a controlled term
in the ontology; for example "acts_on", "acts_during",...
Field Name Data Type Size Default Value Other Foreign Key
id integer 11 PRIMARY KEY, NOT NULL
from_id integer 10 NOT NULL association.id
to_id integer 10 NOT NULL association.id
relationship_type_id integer 10 NOT NULL term.id

Constraints

Type Fields
NOT NULL from_id
FOREIGN KEY from_id
NOT NULL to_id
FOREIGN KEY to_id
NOT NULL relationship_type_id
FOREIGN KEY relationship_type_id

term2term

 Top
 Comments:

**********************************************************
term2term
represents arcs in the graph, (or triple-statements)
If thought of as a graph term1 is parent,
term2 is child.
If thought of as statements, the assertion is
(term2 relationship_type term1)
EXAMPLE:
if term1_id points to nucleic acid binding
term2_id points to DNA binding
relationship_type points to ISA
then we have a statement
"DNA binding ISA nucleic acid binding"
NOTE:
relationship types are kept in the term table, since
these are controlled terms in themselves
Field Name Data Type Size Default Value Other Foreign Key
id integer 11 PRIMARY KEY, NOT NULL
relationship_type_id integer 10 UNIQUE, NOT NULL term.id
term1_id integer 10 UNIQUE, NOT NULL, parent term.id
term2_id integer 10 UNIQUE, NOT NULL, child term.id

Indices

tt4 relationship_type_id

Constraints

Type Fields

FOREIGN KEY relationship_type_id
NOT NULL term1_id
FOREIGN KEY term1_id
NOT NULL term2_id
FOREIGN KEY term2_id
UNIQUE term1_id, term2_id, relationship_type_id

*******************************************************
dbxref
an accession number for an entry in an
external database.
dbxrefs can be used from a variety of places:
as an xref for a sequence
as an xref for a gene product (gene_product != seq except in the case of
GOA)
as an xref for a term (eg swissprot keywords)
as evidence for an association based on sequence evidence
xref_dbname is NOT standardized - it comes from a varierty of sources;
you can treat the following as synonymous
FB FlyBase
WB WormBase
SP SWP SPTR SWALL
if you use a copy of the db loaded via the perl API, then
the dbname will be lowercased to help achieve uniformity
at a later date we will include metadata on the various data
sources
For now, refer to this file:
ftp://ftp.geneontology.org/pub/go/doc/GO.xrf_abbs

** go-optimisations.sql **
** **
** these are OPTIONAL optimisations; they are only **
** suitable for a 'warehouse' style GO database; **
** this is because every time one of the main **
** tables is updated, an update here is forced **
** which would be expensive. **
** **
*******************************************************
Paths between any two terms
(traversing down only)
this table states whether there exists a
path between a parent and a child, and the
distance between them. multiple paths mean
multiple entries in this table
[this table only used in "data warehouse mode"]
an entry also exists linking every term with itself
of distance 0. This is known as "reflexive transitive
